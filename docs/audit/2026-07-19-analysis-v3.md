# LLM-TOP Deep-Dive Analysis (v3 — re-verified against the current tree)

**Method for this pass:** static source review of the working tree
(headers + `.cpp`, tests, CMake, docs). Unlike the v2 pass, I did **not** recompile or execute the
targets in this pass — every claim below is anchored to a specific `file:line` in the current source,
and where I reference test behaviour I describe what the test *asserts*, not a run I performed. The
working directory is not a git checkout, so v2's commit-cadence narrative is unverifiable here and has
been dropped.

This supersedes the v2 report. **Several of v2's central findings are now stale and have been removed**
(see §2) — the crypto is no longer a no-op, `parser.cpp` is gone, key ordering is fixed, and
`decode_statement()` exists. The issues that remain are different and, in the security case, more
structural than the ones v2 was chasing.

### Still open (ranked)

1. **§3.6 — replay / idempotency (top remaining security item).** Capability tokens are still pure bearer credentials: no nonce, proof-of-possession, or idempotency store. `README.md` feature #5 still claims idempotency tags "prevent redundant tool executions" — there is **no** idempotency parsing or store in code. Implement an idempotency-key store, or drop the claim.
2. **§3.5 residual — naive claim parser.** `extract_string_claim`/`extract_int_claim` (`middleware.hpp:269-292`) are first-match string scanners, not a JSON parse. Not exploitable now (escaping + HMAC), but brittle; replace with a real JSON parse for defense-in-depth.
3. **§5 — tolerant-mode self-healing still mutates input.** Mitigated (healed statements quarantined in `healed_draft`; middleware refuses them unless `accept_healed`, `middleware.hpp:319`), but still mutate-then-guess.
4. **§5 residual — recovery *text* instruction.** `generate_recovery_instruction` escapes `reqid` but interpolates raw `error_details` into the LLM-TOP text payload (`fallback_recovery.hpp:98`); low risk (parser-sourced) but unescaped.
5. **Lower-severity §3 — log injection** (raw input echoed to `std::cerr` in tolerant mode) and non-constant-time `iss/aud/scope` compares — not re-verified here; likely still present.
6. **§4.4 / doc honesty.** The "Why LLM-TOP?" anecdote framing and the idempotency claim (item 1) remain in `README.md`.

### Remaining optimizations & follow-ups (notes)

**Performance (§6, still open):**
- `escape_json` builds a fresh `std::ostringstream` **per string** (`json_utils.hpp:10-31`) — the single hottest allocation; a direct `std::string` append with `reserve` is roughly an order of magnitude faster. (`toJson` itself was already improved to `+=` append, so that half of §6 is done.)
- `SHA256::update` copies input one byte at a time — `memcpy` full 64-byte blocks; now matters because CHK hashes the whole body on every `evaluate()`.
- `scope_matches`/`split_scope` re-split both scopes with `std::stringstream`+`getline` on every `verify()` (`middleware.hpp:239-266`).
- Lexer still appends char-by-char over an `istringstream`; a single-pass `string_view` lexer avoids the copies.

**Tokenizer (new, from this slice):**
- BPE merge is O(n²) per pre-token (rescans all adjacent pairs each merge). Fine for short pre-tokens; if large blobs are ever tokenized, port tiktoken's rank-linked-list merge.
- Pre-tokenization is **ASCII-specialized** (`\p{L}/\p{N}/\s` → ASCII subsets): exact for the ASCII payloads benchmarked, approximate for non-ASCII. Extend to UTF-8 if inputs can be non-ASCII.
- Construct `Cl100kTokenizer` once and reuse — it parses the full 100k-entry ranks file at construction (benchmarks already do).

**Build / test infra:**
- `OP_HR` opcode is declared but `encode_header()` still never emits the header `hr` counter — the last dangling bit of §4.3; binary header round-trip is otherwise complete.
- Integration fixtures live **outside** the git repo (`../LLM_Mock`, and `LLM_Mock/` is git-ignored), so `integration_tests` is non-hermetic — it breaks on a fresh clone until fixtures are regenerated. Move them into the repo (e.g. `LLMTOP/test_fixtures/`) and update the test + benchmarker paths.
- A real **sanitizer build is missing**: `build_sanitize/` referenced an `LLMTOP_SANITIZE` option that `CMakeLists.txt` no longer defines, and its flags contained no `/fsanitize`. Re-add an ASan/UBSan option so the fuzzer runs instrumented.

## 1. What it actually is (unchanged)

LLM-TOP is a terse command/control protocol for multi-agent orchestration, **not** a TOON/TRON-style
data-serialization format. A "Planner" emits dense shorthand
(`tgt:src/main.cpp act:refactor GL:fix_memory_leak TD:close_db_connection`); a decoder subagent parses
and executes it; tool calls look like `!read[path=src/db.cpp;cap=<jwt>;ttl=999]`. Authorization metadata
(capability tokens) is bolted onto the dispatch format. Positioning is unchanged since v1.

## 3. Security assessment (current) — most severe first

### 3.1 The middleware is fail-open — the "arbiter" gates nothing without a token `middleware.hpp:327-360`
`evaluate()` validates a KV pair only *if its value contains `cap=`* (`:332`) and a tool call only *if it
carries a `cap` arg* (`:346`). Everything else is skipped, and the function unconditionally ends with
`plan.authorized = true` (`:360`).

```
!read[path=/etc/secrets]     // no cap arg  → never checked → AUTHORIZED
tgt:src/secret.ts            // no cap=      → never checked → AUTHORIZED
```

An attacker bypasses the whole capability system by **omitting the token**. A capability arbiter must
default-deny sensitive/mutating actions and *require* a valid capability; this one only inspects
capabilities that are voluntarily presented. This single design choice defeats the README's
"execution arbiter that cryptographically validates" positioning, regardless of how good the crypto is.
**This is the top issue to fix.**

### 3.2 Tool-call capabilities authorize the tool name, not its arguments `middleware.hpp:345-356`
For tool calls the checked scope is `"execute:" + tool.name` (`:348`). The arguments (`path=…`) are never
bound to the token. A token granting `execute:read` authorizes `!read[path=<anything>]`, including paths
the grantor never intended. Note the inconsistency: KV pointers *do* bind the resource
(`extract_resource_path`, `:395-401`, requires `granted scope == resource`), so the two code paths scope
capabilities at different granularities.

### 3.3 `ttl=` is parsed by nobody — "enforces TTL limits" is not real `middleware.hpp:383-393`
Every example uses `cap=TOKEN;ttl=…`, and the README claims the middleware "enforces TTL limits."
But `extract_capability_token` stops at the first `;` (`:388`), discarding `ttl=`. Nothing reads or
validates it. Expiry rests entirely on the JWT `exp` claim (`middleware.hpp:126-130`); the visible `ttl=`
is decorative. `schema_validator.hpp:176` and recovery hint #4 (`fallback_recovery.hpp:99`) both imply
`ttl=` is checked — it isn't.

### 3.4 The `CHK:sha256:…` header checksum is never verified `parser_v2.hpp:374`, `schema_validator.hpp:144`
The checksum is parsed, stored, echoed into JSON, and *warned about if missing* — but no code computes a
SHA-256 over the payload body and compares. `sha256.hpp` is present and used for HMAC, yet the integrity
field it exists for is unused. The implied tamper-detection is illusory.

### 3.5 JSON injection when minting tokens `middleware.hpp:52-63`
`create_token` concatenates `sub`, `scope`, `iss`, `aud` straight into the payload JSON with **no
escaping**. Combined with `extract_string_claim` returning the *first* match of `"scope":"…"`
(`:267-275`), a caller who controls `sub` but not `scope` can smuggle a broader scope:
`sub = 'x","scope":"execute:*'` yields `{"sub":"x","scope":"execute:*","scope":"execute:read",…}` and the
verifier reads the injected `execute:*`. Privilege escalation at the minting boundary.

### 3.6 Pure bearer model — no replay protection, self-asserted identity
Capability tokens are bearer credentials: no nonce, no proof-of-possession, no channel binding. The
header `AGT` is unauthenticated attacker-controlled text; the `sub == agt` check (`middleware.hpp:376`)
only prevents relabeling a stolen token, not impersonation. README feature #5 claims idempotency tags
"prevent redundant tool executions," but **there is no idempotency parsing or store anywhere** — the word
appears only in prose, not in code. Captured tokens are replayable until `exp`.

### Lower-severity
- Raw attacker input is echoed to `std::cerr` in tolerant mode (`parser_v2.hpp:295`), enabling
  log/newline injection into diagnostics.
- `iss`/`aud`/`scope`/`sub` are compared with non-constant-time `std::string::operator!=`
  (`middleware.hpp:133-145`) — low value since it's post-signature.

### What the crypto layer gets right (credit)
`alg:none` and unknown algs are rejected (`middleware.hpp:97-99`); signature length is checked before
compare (`:106`); the HMAC comparison is constant-time (`sha256.hpp:199-206`); an empty secret throws
rather than silently allowing (`middleware.hpp:46`); healed statements are refused unless explicitly
opted in (`:317-320`); and the parser enforces a max payload size (`parser_v2.hpp:190-193`). The SHA-256
and HMAC implementations themselves look correct against FIPS 180-4 / RFC 2104.

## 4. Fallacies — claims vs. code

### 4.1 The "~53% token reduction" headline is measured with a homemade heuristic, not a tokenizer `benchmarker_real.cpp:9-35`
`estimateTokens()` counts **every punctuation character as its own token** (`:29`) and adds a subword
every 5 characters. Real BPE tokenizers (tiktoken / the Claude tokenizer) merge JSON structure such as
`":"`, `",\n  "`, `"},{"` and collapse whitespace runs into single tokens. The heuristic therefore
over-counts exactly what JSON has most of — structural punctuation — inflating LLM-TOP's advantage by
construction. The baselines are also hand-written *verbose* JSON chosen by the author. The tell is already
in the data: against YAML the same method reports only ~10% median and **as low as −4%**
(`IMPROVEMENTS.md:162,170`). **No "% token reduction" figure in this repo should be published until it is
re-measured with a real tokenizer against a minified-JSON baseline.**

### 4.2 "Perfect semantic fidelity" is contradicted by the serializer `parser_v2.hpp:147-151`
`toJson()` emits only `version` and `checksum` from the header and **drops `agt`, `uid`, `tim`, `reqid`,
`fallback`, `hr`**. The tool's own canonical JSON output loses the agent identity and request ID — not
"perfect fidelity." (Note: this is a *new* fidelity gap; v2's ordering complaint is now moot because
`ordered_map` fixed KV ordering.)

### 4.3 The binary encoder is still lossy — just differently than v2 said `binary_encoder.hpp`
`decode_statement()` now exists, but:
- `encode_statement()` takes a `std::unordered_map` (`:91`), **re-introducing the key-ordering loss** at
  the binary layer that `ordered_map` was added to prevent at the text layer.
- Tool calls are encoded as **names only** ("simplified: just names for now", `:124-127`); tool args and
  methods are never written, so a decoded statement can't reconstruct the call.
- `OP_TTL`/`OP_HR` opcodes are declared (`:40,:31`) but `encode_header()` never emits `HR`.

### 4.4 The "Why LLM-TOP?" anecdote argues against itself `LLMTOP/README.md:42-45`
It says subagents "bridged massive semantic gaps" to parse `GL:AStar_pathfind`. If the model must bridge a
massive gap, the compression is **lossy in meaning** and depends on the model guessing intent — the
opposite of the "deterministic / perfect fidelity" framing. It is an unfalsifiable claim from "simulated
Phase 3 stress tests" with no data, and it ignores prompt overhead (the format must be taught to the
model) and the fact that dense non-English abbreviations often tokenize *worse* per unit of meaning than
plain English.

## 5. Correctness bugs

- **`decode_varint` can overflow** `binary_encoder.hpp:257-272`: on the 5th byte,
  `(byte & 0x7F) << 28` can exceed 32 bits and wrap. Not exploitable (downstream bounds checks throw), but
  it decodes wrong values for large varints.
- **`error_count` is effectively boolean** `fallback_recovery.hpp:43-46`: diagnostics are concatenated
  into one string, so `errors_found` is only ever 0 or 1 no matter how many errors occurred — the recovery
  report's counts mislead.
- **Unescaped fields in recovery output** `fallback_recovery.hpp:82-83,116-117,154,160`: `reqid`, `agt`,
  `role`, and `kv.first` are interpolated into JSON / an LLM-TOP payload without `escape_json`, so an
  attacker-controlled `REQID` containing `"` can break or inject into the fallback structure that
  downstream agents consume.
- **Tolerant-mode "self-healing" mutates input** `parser_v2.hpp:233-263`: appending a `"` or inserting a
  `]` can silently change a line's meaning. Statements are correctly quarantined as `healed` (good), but
  mutate-then-guess is risky for a security-adjacent parser.

## 6. Performance

Relevant because the project markets itself for "scalable, high-throughput autonomous systems":

- **`escape_json` builds a fresh `std::ostringstream` per string** (`json_utils.hpp:10`), called for every
  key and value in `toJson`. `ostringstream` construction/locale is among the slowest ways to build a
  string in C++; a direct `std::string` append with `reserve` is roughly an order of magnitude faster.
- **`toJson` concatenates via chained `operator+`** (`parser_v2.hpp:147-177`), creating many temporaries
  per statement despite the up-front `reserve`.
- **`SHA256::update` copies input one byte at a time** (`sha256.hpp:31-40`) instead of `memcpy`-ing full
  blocks — fine for tiny JWTs, poor for the "payload checksums" the header comment advertises.
- **The lexer appends char-by-char** into per-token `std::string`s over an `std::istringstream`
  (`parser_v2.hpp:299-329`) — lots of small allocations; a single-pass `string_view` lexer would avoid the
  copies.
- `scope_matches` re-splits both scopes with `std::stringstream`+`getline` on every verify
  (`middleware.hpp:237-264`).

## 7. Bottom line

Between v2 and now, the crypto went from theatre to real: HMAC-SHA256 is implemented and correct, `alg:none`
is rejected, the signature compare is constant-time, and base64url round-trips. The dead `parser.cpp` is
gone and key ordering is fixed. Those are genuine improvements and v2's findings on them are retired.

But the security *architecture* around the good crypto is the problem now: the middleware is **fail-open**
(§3.1) — omit the token and you're authorized — tool-call capabilities don't bind their arguments (§3.2),
the advertised `ttl=` and `CHK` integrity fields are never enforced (§3.3, §3.4), and token minting is
JSON-injectable (§3.5). Separately, the flagship "~53% token reduction" rests on a hand-rolled token
*estimator* rather than a real tokenizer (§4.1), and `toJson` quietly drops most header fields (§4.2),
undercutting the "perfect fidelity" claim.

The through-line, same as both prior passes: **a claim needs a test that would fail if the claim were
false.** Today the security tests exercise only the happy path where a token is present, so the fail-open
gap is invisible; and the compression claim is validated by an estimator that can't be wrong in JSON's
favour. Fix those two testing gaps and the two headline claims (crypto-enforced authorization, token
savings) would either hold up or visibly break.

### Priority fixes

> **✅ All six below are complete as of 2026-07-19 — see §0 for per-item evidence. The main remaining
> work is §3.6 (replay/idempotency) and the §6 performance items.**

1. Make `LLMTOPMiddleware::evaluate` **default-deny**: require a valid capability for every tool call and
   every mutating pointer; add a test with a token-less payload asserting `authorized == false`. *(§3.1)*
2. Bind tool-call capabilities to their **arguments/target**, not just the tool name. *(§3.2)*
3. Either **verify `CHK`** and enforce `ttl=`, or delete them and stop claiming integrity/TTL enforcement.
   *(§3.3, §3.4)*
4. **Escape all interpolated fields** in `create_token` and the recovery serializers. *(§3.5, §5)*
5. Re-run the benchmark with a **real tokenizer + minified-JSON baseline** before quoting any percentage.
   *(§4.1)*
6. Round-trip the full header in `toJson`; encode tool args/methods and preserve key order in the binary
   encoder. *(§4.2, §4.3)*
